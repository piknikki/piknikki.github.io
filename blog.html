A new app deployed!

Nikki: 1, Heroku: 0

Date: July 17, 2019

I shouldn't say that I won over Heroku, because when I can get my shit deployed, we all win (even Heroku). But holy crap can it be a struggle to get things to work when deploying!

So I guess this post should be about problem solving and grit. I have a lot of feelings about grit, and I wrote about them on a post on LinkedIn, so let's focus on problem solving here.

I was using a tutorial to learn some Redux to go with my React knowledge, and I had completed the entire course. It took me about 2 weeks, from start to finish. I learned a lot, solidified knowledge that I learned a while back, and got some new tips/tricks from the developer teaching the course. It was muy bueno. But then . . . Heroku. ü§™ ü§™ ü§™

The first problem was that I was unfamiliar with a post-production script build instead of a local build and then pushing the build. So that was fun, to learn how easy it is to do that -- just add a script in the package.json, make sure your secrets are tightly kept in a config and .gitignore situation, and then push it! Wheee! (that's really not all; there are other changes that need to be made, but I'm too tired to put them all down; like making sure your process.env.NODE_ENV is set to 'production'; etc. etc.)

I pushed to my master and to Heroku's master, the build was confirmed, and Heroku says . . . no. üôÖ

Why, Heroku? Why you do me this way?? ü•∫

When this happens, you need to look, listen, and ask questions. Look at the Heroku logs or any other errors you may be getting. Listen to what google and stack overflow are saying. If you look at the errors but don't know how to solve them, ask experts! Use google and stack overflow because it is pretty much guaranteed that someone else has had the *exact* same problem before -- or at least something close enough that you can adapt a solution to your problem.

So my last step in this problem solving process was to be patient with Heroku. It takes a while to finish its biz when you've made changes. Give it time to do the magic. And when I say time, I mean more than 60 seconds, which is like, f o r e v e r when you're trying to diagnose a problem with deploying! I know. I feel your pain. But try to be patient. I had made some fixes, and tried reloading but it wasn't working. 5 minutes later, I tried a refresh and there was my app. Magic.

The problem with that is that I don't know which solution ended up fixing the problem. Was it adding node and npm versions to the package.json file? Or was it adding the variables to Heroku's config? The answer is, probably all the things are the solution. But I wasn't patient enough to let them percolate through Heroku. üòù ü§Ø üò© üò§ ü§¨ üôè üôå üíÜ

GitHub pdf for collaborating (read and download)

A somewhat helpful guide??

Date: June 28, 2019

Click on the image below to go to my GitHub and then click on "download" in the upper right.


In preparation for teaching a new tutoring student about how to collaborate with GitHub, I created a pdf of a flow chart for working with others and how GitHub works. This is the description that goes with that instructional and super helpful pdf.

Ok, on the first page, I have "my repo" in purple-ish blue. It shows how, using the git commands, I can do my own git version control in my repo. I can create a repo on GitHub and go from there, or I can create one locally, then add the remote and push up to my GitHub.

On the second page, you can see your repo, which is, you guessed it, the same thing! You can do your own version control with git in your own repo. Yay!

The last page shows how we would collaborate together, assuming that I create (and thus own) the original repo. You could either clone or fork my repo: cloning copies the repo to your local machine, whereas forking copies the repo to your GitHub. With the former scenario, your local copy is intrinsically your working directory and your local repo; you would make changes and then you could create a pull request (after add, commit, and push). With the latter scenario, your copy is on GitHub, so you'll need to clone it to your local machine in order to work on it. Then, as with cloning, you can add, commit, push, and then make a pull request. This all assumes that the final product will be on my repo (my GitHub).

Let's say that you wanted to keep it to yourself. For example, I've worked on projects with a group, and after we're done and deployed, and they wanted to make changes and deploy to their own Heroku or other platform. Cool. Then you would just fork my repo and go from there on your own GitHub without creating a pull request. Easy peasy! üç©

Testing -- Bug lifecycle and a bunch of testing concepts

My first basic learning steps in testing

Date: June 27, 2019

I've starting learning some things about testing, and want to write about what I've learned (to reinforce the learning).

Bug Lifecycle: This is pretty self-explanatory once you look at it. Obviously, someone needs to report a bug, and that's when it gets put into a tracking system as "new". Then it is assigned to a developer, who changes the status to "active" if they believe it is truly a bug, and "rejected" if they don't. The latter may be the case if a QA Tester doesn't understand the requirements or if the feature is working, just not as the tester expected. If the bug is accepted and active, the developer works on a fix and then tests the feature. If it fails at this point, the bug can be reopened and goes back into the flow as "active"; if it passes, the status is changed to "verified" and then it's closed.

Black box testing: this type of testing is performed when the tester does not know the "inner workings" of the feature. Examples of black box testing include: equivalence testing (representatives from each partition are selected; an additional note is that the tester will want to confirm both positive and negative test cases (i.e. valid and invalid cases), boundary testing (edge cases, and cases on either side of the partitions), decision tables (created using the test conditions to build a matrix of possible outcomes), state transition (when the feature changes state, like login/logout), and use cases (specific scenarios).

White box testing: this type of testing is performed when the tester understands and knows the inner code and how the feature is built by the developer. Examples of white box testing include unit testing and statement testing.

Smoke & Sanity checks: Smoke testing tests the basic and most important functionality of a feature or site, and can be a major cost saver because it gets the MVP features tested right away. Sanity checks are just what it sounds like: does this make sense??

Regression testing: aka verification testing. This is used to confirm that bug fixes that have been made haven't broken other parts of the code or feature.

Other testing concepts: Risk-based testing basically assesses risks and gets people with different perspectives involved in asking questions about risk. Static testing assesses functionality of the software without running it (and might even be considered "code review", whereas dynamic testing runs the software (executes the code). Static is used earlier in the development life cycle, and dynamic is used later.

As a last thought here, it should be fairly obvious that planning testing and estimation should be just as detailed as planning the development of software itself. What to test, how to test, how long testing should take, etc. will all need to be thought through early (and be adjusted as necessary throughout the process). This is where scrum and agile can help out, using techniques to estimate the complexity and energy required to complete a task (not necessarily time or people, but what it will take to get something done -- and what the definition of "done" actually is!). ü•û ü•ì

Tackling a Topic one thought at a time

HTML, CSS, and JS

Date: June 18, 2019

This discussion is about the basic trinity of web development: HTML, CSS, and JavaScript. HTML is where the journey begins; it is the structure or the scaffolding on which we build a web page or an app.

HTML is, of course, an acronym. Side note: learning to code is no different from learning medicine, in that it is basically learning a new language. Every new language has its own precarious set of acronyms. HTML stands for Hypertext Markup Language, and it consists of tags that determine what will be placed on the page in the browser. It's not fancy, but it's the backbone of anything you see on the web; it is the starting place for any and all websites, web pages, and web applications. HTML may seem simple, but it is essential for any web page to "work" (i.e. be seen by the user).

CSS, (also an acronym) stands for Custom Styling Sheets, and it is how we create a bit of zhoosh for the HTML we have. CSS is the sparkle, the glitter, and the icing on the cake. It allows us to place elements in special locations on the page, change colors and fonts, and basically transform boring cake into a work of art. To learn more about CSS, I have delved into a few different ways to use it, like SCSS and BEM. I'll profile those in another article.

JavaScript (JS) is where things start to get real (and complex). JS allows users to interact with the web page or application (HTML and CSS create static content -- it's not interactive). JS makes buttons clickable, allows content to be dynamically updated, and animates images.

If we go a step beyond the "holy trinity", we find APIs (Application Programming Interface). APIs are the zhoosh to JavaScript -- they create extra power and functionality to JS by using additional code. It's like borrowing someone else's Legos to build your house (but the Legos are reusable and can be shared around the world).

Ok, this was just a basic surface-scratching post, to get things started. My creative juices are in desperate need of refreshing, because technical writing is turning out to be harder than I had anticipated it would be.

Onward!!
