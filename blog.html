Testing -- Bug lifecycle and Black box/White box testing

My first basic learning steps in testing

Date: June 27, 2019

I've starting learning some things about testing, and want to write about what I've learned (to reinforce the learning).

Bug Testing: This is pretty self-explanatory once you look at it. Obviously, someone needs to report a bug, and that's when it gets put into a tracking system as "new". Then it is assigned to a developer, who changes the status to "active" if they believe it is truly a bug, and "rejected" if they don't. The latter may be the case if a QA Tester doesn't understand the requirements or if the feature is working, just not as the tester expected. If the bug is accepted and active, the developer works on a fix and then tests the feature. If it fails at this point, the bug can be reopened and goes back into the flow as "active"; if it passes, the status is changed to "verified" and then it's closed.

Black box testing: this type of testing is performed when the tester does not know the "inner workings" of the feature. Examples of black box testing include: equivalence testing (representatives from each partition are selected), boundary testing (edge cases, and cases on either side of the partitions), decision tables (created using the test conditions to build a matrix of possible outcomes), state transition (when the feature changes state, like login/logout), and use cases (specific scenarios).

White box testing: this type of testing is performed when the tester understands and knows the inner code and how the feature is built by the developer. Examples of white box testing include unit testing and statement testing.



Scrum & Agile are what floats my boat

In my position leading a nonprofit, that meant finding board members and then giving the everything they needed in order to be successful, from helping them define goals to overcoming challenges in achieving those goals. In parenting, that means telling my kids that they can do something and then letting them do it! Servant Leadership involves trust, respect, and empathy; it takes patience, creates space for mistakes and learning from those mistakes, and relies on the assumption that we are all doing our best.

Doesn't that sound awesome??

Tackling a Topic one thought at a time

HTML, CSS, and JS

Date: June 18, 2019

This discussion is about the basic trinity of web development: HTML, CSS, and JavaScript. HTML is where the journey begins; it is the structure or the scaffolding on which we build a web page or an app.

HTML is, of course, an acronym. Side note: learning to code is no different from learning medicine, in that it is basically learning a new language. Every new language has its own precarious set of acronyms. HTML stands for Hypertext Markup Language, and it consists of tags that determine what will be placed on the page in the browser. It's not fancy, but it's the backbone of anything you see on the web; it is the starting place for any and all websites, web pages, and web applications. HTML may seem simple, but it is essential for any web page to "work" (i.e. be seen by the user).

CSS, (also an acronym) stands for Custom Styling Sheets, and it is how we create a bit of zhoosh for the HTML we have. CSS is the sparkle, the glitter, and the icing on the cake. It allows us to place elements in special locations on the page, change colors and fonts, and basically transform boring cake into a work of art. To learn more about CSS, I have delved into a few different ways to use it, like SCSS and BEM. I'll profile those in another article.

JavaScript (JS) is where things start to get real (and complex). JS allows users to interact with the web page or application (HTML and CSS create static content -- it's not interactive). JS makes buttons clickable, allows content to be dynamically updated, and animates images.

If we go a step beyond the "holy trinity", we find APIs (Application Programming Interface). APIs are the zhoosh to JavaScript -- they create extra power and functionality to JS by using additional code. It's like borrowing someone else's Legos to build your house (but the Legos are reusable and can be shared around the world).

Ok, this was just a basic surface-scratching post, to get things started. My creative juices are in desperate need of refreshing, because technical writing is turning out to be harder than I had anticipated it would be.

Onward!!
